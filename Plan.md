# План автоматизации тестирования веб-сервиса покупки туров

## 1. Перечень автоматизируемых сценариев

### Позитивные сценарии (для формы «Купить» и «Купить в кредит»)
1. **Успешная дебетовая оплата тура картой APPROVED:** 
   * Номер карты: `4444 4444 4444 4441`
   * Остальные поля заполнены валидными данными.
   * *Ожидаемый результат:* На UI появляется всплывающее окно об успешной оплате. В СУБД в таблице `payment_entity` создается запись со статусом `APPROVED`.
2. **Успешное оформление кредита по карте APPROVED:** 
   * Номер карты: `4444 4444 4444 4441`
   * Остальные поля заполнены валидными данными.
   * *Ожидаемый результат:* На UI появляется всплывающее окно об одобрении кредита. В СУБД в таблице `credit_request_entity` создается запись со статусом `APPROVED`.

### Негативные сценарии (для формы «Купить» и «Купить в кредит»)
3. **Отказ в операции по заблокированной карте DECLINED:** 
   * Номер карты: `4444 4444 4444 4442`
   * *Ожидаемый результат:* На UI появляется окно-предупреждение об отказе в операции. В СУБД создается запись со статусом `DECLINED`.
4. **Отправка полностью пустой формы:** 
   * Все поля ввода оставляются пустыми.
   * *Ожидаемый результат:* Форма не отправляется, под каждым полем загорается красная ошибка валидации.
5. **Невалидный формат номера карты:** 
   * Короткий номер карты (15 цифр).
   * *Ожидаемый результат:* Под полем «Номер карты» появляется ошибка «Неверный формат».
6. **Невалидное значение месяца:** 
   * Значение месяца `00` или `13`.
   * *Ожидаемый результат:* Под полем «Месяц» появляется ошибка «Неверно указан срок действия».
7. **Истекший срок действия карты (Прошедший год):** 
   * Год меньше текущего (например, прошлый год).
   * *Ожидаемый результат:* Под полем «Год» появляется ошибка «Истёк срок действия карты».
8. **Избыточный срок действия карты (Далекое будущее):** 
   * Год на 6 лет больше текущего.
   * *Ожидаемый результат:* Под полем «Год» появляется ошибка «Неверно указан срок действия».
9. **Невалидные данные в поле «Владелец»:** 
    * Ввод имени на кириллице, цифрами или спецсимволами.
    * *Ожидаемый результат:* Под полем «Владелец» появляется ошибка «Неверный формат».
10. **Невалидный формат CVC/CVV кода:** 
    * Ввод 2 цифр вместо 3.
    * *Ожидаемый результат:* Под полем «CVC/CVV» появляется ошибка «Неверный формат».

## 2. Перечень используемых инструментов с обоснованием выбора

1. **Язык программирования Python 3.14:** Обеспечивает высокую скорость написания автотестов и отличную читаемость кода фреймворка благодаря лаконичному синтаксису.
2. **Тестовый фреймворк Pytest:** Выбран из-за удобного механизма фикстур (удобно чистить базу данных перед тестами) и встроенной поддержки параметризации.
3. **Инструмент автоматизации Selenium WebDriver:** Индустриальный стандарт автоматизации действий в браузере, позволяющий реализовать паттерн архитектуры **Page Object Model**.
4. **Библиотека SQLAlchemy + драйвер psycopg2-binary:** Инструмент для отправки прямых проверочных SQL-запросов к СУБД PostgreSQL из Python-кода для сверки статусов операций.
5. **Система отчетности Allure Report:** Позволяет генерировать детальные интерактивные отчеты с пошаговым логированием выполнения сценариев и визуальными графиками для экспертов.

## 3. Перечень и описание возможных рисков при автоматизации

1. **Риск изменения интерфейса приложения:** В верстке веб-сервиса отсутствуют уникальные тестовые атрибуты элементов (например, `data-test-id`). При малейшем изменении структуры страниц относительные XPath-локаторы могут сломаться.
2. **Риск асинхронности взаимодействия приложения и СУБД:** Запись данных бэкендом в базу данных PostgreSQL может происходить с микрозадержками, что способно приводить к ложным падениям автотестов (*Flaky tests*). Риск локализован применением паттерна Database Polling (умного цикла ожидания записи).
3. **Риск доступности сторонних компонентов:** Тесты зависят от стабильности работы Docker-контейнеров с симулятором банковских шлюзов и базой данных. В случае зависания контейнеров прогон тестов заблокируется.

## 4. Интервальная оценка с учётом рисков в часах

* Изучение ТЗ, кода сервиса и структуры таблиц БД: **4–6 часов**
* Настройка инфраструктуры (Docker Compose, зеркала образов) и окружения Python: **6–8 часов**
* Проектирование архитектуры Page Object фреймворка и класса проверок БД: **10–12 часов**
* Написание автотестов (позитивные и негативные сценарии) и интеграция шагов Allure: **12–16 часов**
* Оформление баг-репортов в Issues и итоговой документации (Report, Summary): **4–6 часов**
* **Итоговая интервальная оценка:** от **36** до **48** часов.
